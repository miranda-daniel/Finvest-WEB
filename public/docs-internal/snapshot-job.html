Snapshot Job

Daily closing price ingestion for portfolio instruments and benchmarks — Finvest API

The portfolio performance chart (Value and Performance tabs) needs historical daily closing prices for every held instrument. Fetching these from TwelveData on every page load would be slow, expensive, and would quickly exhaust the free-tier API quota.

The snapshot job solves this by running once per day and persisting the prices in the PriceSnapshot table. The GraphQL resolver for portfolioPerformance then reads purely from the DB — no live API calls needed at query time.

When it runs

TriggerScheduleNotes
Cron 0 23 * * * — 23:00 UTC daily 20:00 Argentina time. US markets close at 16:00 ET (21:00 UTC) so all closing prices are available by then.
On startup Immediate, production only Catches up any gap if the server was down. Disabled in development to avoid burning API quota on every restart.

What gets fetched

The job processes all instruments currently held in any portfolio, plus the two benchmark ETFs. Each instrument is handled independently — one instrument's failure does not stop the rest.

📊 Portfolio instruments
All distinct instruments that appear in at least one Holding record, across all users. Fetched via HoldingRepository.findDistinctInstrumentIds().
📈 SPY (S&P 500 proxy)
SPDR S&P 500 ETF. Used as a benchmark instead of the SPX index symbol because TwelveData only exposes index symbols on paid plans. Flagged as isBenchmark = true in the Instrument table.
📈 QQQ (Nasdaq 100 proxy)
Invesco QQQ ETF. Same rationale as SPY — proxy for NDX. Both ETFs track their respective indices very closely and are available on the free tier.

Date range logic — per instrument

For each instrument the job computes the missing date range independently, so only truly absent data is fetched.

Start date

latest snapshot exists → latest snapshot date + 1 day
no snapshots, isBenchmark → 1 year ago (bootstrap full history for the chart)
no snapshots, portfolio instrument → earliest BUY date for that instrument (or 1 year ago if no operations found)

End date

Always the last trading day (Mon–Fri). If the job runs on a Sunday, end date is the previous Friday. This prevents calling TwelveData for a weekend date that has no market data, which would return an error.

today = Sunday → Friday (2 days back)
today = Monday → Friday (3 days back)
today = Tue–Sat → yesterday (1 day back)

If startDate > endDate, the instrument is already up to date and the API call is skipped entirely.

Rate limiting

TwelveData's free plan limits API usage. A 200 ms delay is inserted between each instrument that actually made an API call. Instruments that were skipped (already up to date) do not count — the sleep is only applied when a real request was made or attempted (including errors, to avoid hammering a rate-limited endpoint on retries).

The last instrument in the list does not get a trailing sleep.

Running manually

The job can be triggered on demand without waiting for the 23:00 UTC cron, using the npm run snapshot script defined in package.json:

npm run snapshot

This invokes scripts/run-snapshot-job.ts via ts-node with path aliases resolved, using the environment variables from .env. The process exits cleanly when the job finishes.

🔁 Missed nightly run
If the server was down when the cron fired, run the script manually to catch up. It will fetch all missing dates since the last stored snapshot.
➕ New holdings added
After adding instruments that have no snapshot history yet, run the script to populate the performance chart immediately rather than waiting until tonight.
🔍 Verification
Useful to confirm the job works end-to-end after a configuration change or deployment. Safe to run multiple times — existing rows are updated, not duplicated.

How prices are stored

Prices are written to the PriceSnapshot table via a single INSERT … ON CONFLICT DO UPDATE statement (bulk upsert using Prisma.$executeRaw). The unique constraint is on (instrumentId, date) — re-running the job for the same date is safe and simply overwrites the existing price.

ColumnTypeDescription
instrumentIdIntFK to Instrument
dateDateTrading day (no time component)
closePriceDecimalAdjusted closing price in USD
createdAtDateTimeRow creation timestamp

Who reads these snapshots

PortfolioPerformanceService.getPerformance() is the sole consumer. It queries PriceSnapshot for the requested date range, computes the portfolio value per day by multiplying each holding's quantity (derived from BUY/SELL operations up to that day) by the closing price, then normalises all three series (portfolio, SPY, QQQ) to a common base of 0% on the first day with a non-zero portfolio value.
