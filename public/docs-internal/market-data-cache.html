Market Data Cache Flow

How Redis caches TwelveData API calls for live quotes and EOD prices — Finvest API

TwelveData's free plan (Basic 8) allows 8 API credits per minute. Each symbol in a batch request consumes 1 credit, so a portfolio with 10 holdings would spend 10 credits per call — exceeding the limit on a single page load that requests both live quotes and EOD prices. Redis eliminates redundant calls by caching responses close to the API layer.

Request flow

🌐 Frontend (React)
GET /api/instruments/quotes?symbols=…
🟢 InstrumentController
InstrumentService.getBatchQuotes(symbols)
⚙️ InstrumentService
redis.get("quotes:ACN,NVO,TSM")
cache
hit?
HIT ✓ ──→
return cached JSON immediately
MISS ✗ ──→
🌍 TwelveData /price API
result → redis.setex(key, ttl, JSON)
🔴 Redis (store with TTL)
return prices to controller → HTTP response
🌐 Frontend receives prices

The same flow applies to GET /api/instruments/eod — the only difference is the cache key and TTL strategy.

Cache keys & TTL strategy

Endpoint Redis key pattern TTL Rationale
Live quotes
/instruments/quotes		 quotes:ACN,NVO,TSM
(symbols sorted, comma-joined)		 60 seconds Prices change constantly, but a 60s delay is acceptable for personal portfolio tracking. Short enough to feel live, long enough to absorb repeated reloads.
EOD prices
/instruments/eod		 eod:ACN,NVO,TSM:2026-05-02
(symbols + ISO date)		 Until midnight + 1h buffer Previous close never changes within a trading day. The date in the key acts as natural invalidation — the next calendar day produces a different key, so TwelveData is called once per day per symbol set regardless of how many requests come in.

Implementation details

📂 src/cache/redis.ts
ioredis client singleton. Created once at startup with lazyConnect: true so it doesn't block startup if Redis is momentarily unavailable. Connection errors are logged but do not crash the process.
⚙️ src/services/instrument-service.ts
Cache logic lives in the service layer — not in the client. The client (twelve-data-client.ts) remains a pure HTTP wrapper. The service checks Redis before calling the client, and writes the result back to Redis after a cache miss.
🔴 Redis in Docker
Defined in docker-compose.dev.yml as redis:7-alpine. Port 6379 exposed to the host. The API container depends on it (health check via redis-cli ping) and receives REDIS_URL as an env variable.
🚨 TwelveData error detection
TwelveData returns rate-limit and other errors as HTTP 200 with a JSON body containing status: "error". The client detects this and throws, so the error propagates instead of being silently cached as {}.
🔮 Future uses of Redis
Redis is already in the stack — future use cases include: rate limiting (express-rate-limit backend for auth endpoints), refresh token storage (replace DB rows with Redis keys + TTL), and job queues (BullMQ for background processing).
