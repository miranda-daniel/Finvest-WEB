Each F5 rotates the refresh token — the old one is revoked and a new one is issued. Active Sessions always shows at most one entry per device.

Logout

The backend revokes the refresh token and clears the cookie. The frontend clears Zustand and the Apollo cache. onSettled is used instead of onSuccess so auth state is always cleaned up even if the request fails.

6
Refresh Token Theft Detection

Each refresh token can only be used once. If a token that was already rotated (and therefore revoked) is used again, the backend assumes it was stolen and wipes all sessions for that user.

Attacker
API
Database
POST /api/session/refresh-token (old, already-rotated cookie)
SELECT RefreshToken WHERE token = hash
found — but stored.revoked ≠ null → THEFT DETECTED
UPDATE all tokens WHERE userId = X SET revoked = now()
401 INVALID_REFRESH_TOKEN
⚠ All sessions for that user are now revoked — including legitimate ones. The real user will be forced to log in again on all devices. This is intentional: if a stolen token was used, the account may be compromised.
7
The silentRefresh Singleton

Three independent callers can trigger a refresh simultaneously: the route guard, the Axios response interceptor, and the Apollo error link. Without coordination, each would fire its own HTTP request — creating multiple sessions and leaving orphaned (non-revoked) tokens.

src/api/silentRefresh.ts — shared in-flight promise
❌ Without singleton
Route guard → POST /refresh-token → new session A
Axios interceptor → POST /refresh-token → new session B
Apollo error link → POST /refresh-token → new session C
Sessions A and B are never revoked → leaked tokens
✓ With singleton
Route guard → silentRefresh() → starts POST /refresh-token
Axios interceptor → silentRefresh() → inFlight exists → waits
Apollo error link → silentRefresh() → inFlight exists → waits
All three resolve with the same new JWT. One session created.
