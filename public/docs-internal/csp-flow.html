Qué es cada pieza

🪖 Helmet

Un middleware de Express que agrega headers de seguridad HTTP a todas las respuestas del servidor. Uno de esos headers es el CSP.

app.use(helmet())

🔒 CSP (Content Security Policy)

Un header HTTP que le dice al browser desde qué dominios puede cargar recursos (scripts, imágenes, etc.). Si un dominio no está en la lista, el browser lo bloquea — sin importar si internet funciona.

Content-Security-Policy:
script-src 'self' cdn.jsdelivr.net

🌐 Apollo Sandbox

La UI interactiva que aparece en /graphql. No viene bundleada con el servidor — el browser la descarga desde el CDN de Apollo cada vez que la abrís.

embeddable-sandbox.cdn
.apollographql.com/v2/...

📦 CDN

Un servidor de archivos estáticos distribuido globalmente. Apollo aloja su Sandbox en un CDN de Google Cloud para que cargue rápido desde cualquier parte del mundo.

cdn.apollographql.com
→ IP: 34.120.73.219

Flujo con el bug

1
Browser abre http://localhost:3001/graphql
Hacés GET a la ruta de GraphQL en tu servidor local.
2
Apollo Server responde con el HTML del Sandbox Apollo
El servidor devuelve una página HTML mínima. Esa página incluye un <script> que apunta al CDN de Apollo para cargar la UI real.
<script src="https://embeddable-sandbox
.cdn.apollographql.com/v2/..."></script>
3
Helmet agrega el header CSP en la respuesta Helmet
Junto con el HTML, el servidor agrega headers de seguridad. El CSP solo permite scripts de 'self' y cdn.jsdelivr.net.
Content-Security-Policy:
script-src 'self' cdn.jsdelivr.net
connect-src 'self' cdn.jsdelivr.net
4
Browser lee el CSP y bloquea el script BLOQUEADO
El browser ve que el script viene de apollographql.com, que NO está en el CSP. Lo bloquea antes de hacer el request. Ni siquiera llega a la red.
❌ Refused to load script from
'embeddable-sandbox.cdn.apollographql.com'
because it violates CSP directive:
script-src 'self' cdn.jsdelivr.net
5
La UI no puede cargar → mensaje de error
Sin el script del CDN, la página no puede inicializar el Sandbox y muestra el mensaje confuso.
Por eso el curl funcionaba (200 OK) pero el browser mostraba error:
curl no ejecuta JavaScript ni aplica CSP — hace el request directamente. El browser sí aplica el CSP y bloqueó el script antes de que llegara a la red.

Flujo corregido

1
Browser abre http://localhost:3001/graphql
Igual que antes.
2
Apollo Server responde con el HTML del Sandbox
Igual que antes — el HTML incluye el script del CDN de Apollo.
3
Helmet desactiva el CSP en desarrollo Helmet
El Sandbox carga scripts, fonts, manifests y favicons desde múltiples subdominios de Apollo. Agregar cada uno al CSP manualmente es frágil — si Apollo agrega un recurso nuevo, volvería a romperse. La solución es deshabilitar el CSP solo en desarrollo.
// index-middlewares.ts
const devHelmetConfig = {
contentSecurityPolicy: false, // deshabilitado en dev
};

app.use(helmet(
isProduction() ? undefined : devHelmetConfig
));
4
Browser descarga todos los recursos del Sandbox ✓
Sin CSP que lo restrinja, el browser puede cargar libremente los scripts, estilos, fonts y manifests que el Sandbox necesita.
✅ embeddable-sandbox.cdn.apollographql.com → 200 OK
✅ fonts.googleapis.com → 200 OK
✅ manifest.json → 200 OK
5
Apollo Sandbox carga correctamente ✓
La UI inicializa. Podés explorar el schema, ejecutar queries y mutations directamente desde el browser en http://localhost:3001/graphql.
El fix fue deshabilitar el CSP de Helmet solo en desarrollo.
El Sandbox carga demasiados recursos externos para mantener una whitelist manualmente. En producción el Sandbox está deshabilitado y el CSP restrictivo sigue aplicando sin cambios.
